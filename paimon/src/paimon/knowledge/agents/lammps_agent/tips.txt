6. Post-Processing and Finalization
After simulation completes, convert thermodynamic outputs to .h5 format.

```python
from simulation_util.lammps import write_lammps_thermo_to_hdf5
from simulation_util.lammps.data import resolve_from_input

# You should write a detailed run_note for each run you have performed. This includes output frequency, simulation time, and timestep.
run_note = "NPT at 298 K and 1 bar; time step 2 fs, trajectory saved every 100 steps, thermodynamic output saved every 10 steps, simulation time 1 ns"

# The topology (lammps-data) file should be resolved from the LAMMPS input script.
topology_file = resolve_from_input("lammps.in")  # returns path to original lammps-data. e.g. ../01_prev_lammps_run/AcU.lammps-data
print(f"Resolved data path: {topology_file}")

# Auto-writes all runs from log. If the log contains multiple sequential runs, run_note must be a list of strings in order.
write_lammps_thermo_to_hdf5(
    log_path="npt.log",
    output_h5="data.h5",
    type_map=["Ac", "U"],  # type_map to map lammps type to element (mandatory) 
    topology_file=topology_file,
    trajectory_file="./traj.dcd",             # Path to trajectory
    run_note=run_note,        # Per-run note, str | list of str
)
```

For multiple independent simulations (replicates), you can loop over log files:

```python
        random_seed=seed, trajectory_file=f"./traj_{seed}.dcd"
    )  # run_note is omitted for brevity
# Result: replicates.h5 with run_0, run_1, run_2
```

<lammps script parsing rules>
<prefixes>
`c_`, `v_`, `f_` reference values produced by computes, variables, or fixes.
These force runtime evaluation, not input-parse substitution.

- `c_ID`: output of `compute ID ...`
- `v_name`: output of `variable name ...`
- `f_ID`: output of `fix ID ...`

Valid only after the corresponding compute/variable/fix is defined.
Note: These prefixes are reference-only syntax, not part of the definition. Never include them in the ID (compute/fix) or name (variable) itself (e.g., variable myvar equal ... → reference as v_myvar; compute myke ... → reference as c_myke).
</prefixes>

<parsing>
- `${var}`: variable substitution only. v_ prefix must not be included inside. Substituted once at input parse time.
- `v_`, `c_`, `f_`: evaluated when the command is executed. Accepted by commands that support runtime evaluation (e.g. thermo_style) or in a immediate experssion. 
- c_ID[i], v_name[i], f_ID[i]: Specify indexes for vectors. [*] refers to all elements.
- $(...): evaluates formula immediately as an equal-style variable expression. Unlike ${}, prefixes (v_, c_, f_) can be used inside.
</parsing>

<immediate_expressions>
$(...) evaluates the enclosed formula immediately as an equal-style variable expression,
without defining a named variable.
  ```
  variable X equal (xlo+xhi)/2+sqrt(v_area)
  region 1 block $X 2 INF INF EDGE EDGE
  variable X delete
  ```
is equivalent to:
  ```
  region 1 block $((xlo+xhi)/2+sqrt(v_area)) 2 INF INF EDGE EDGE
  ```
</immediate_expressions>

<predefined_variables>
The following keywords can be used directly in equal/vector/atom-style `variable` formulas and in $(...) immediate expressions.
lthough these act as variables, v_ prefix is not required when referencing them.
For example, xlo and xhi is a predefined variables:
e.g. region 1 block $(xlo+1.0) $(xhi-1.0) INF INF EDGE EDGE

  <timestep>
    dt: timestep size (time units)
    time: elapsed simulation time (time units)
  </timestep>

  <system>
    atoms: number of atoms
    temp: temperature
    press: scalar pressure
  </system>

  <energy docs="compute_pe">
    pe: total potential energy
    ke: kinetic energy (derived from temp)
    etotal: total energy (pe + ke)
    enthalpy: enthalpy (etotal + press*vol)
  </energy>

  <box>
    vol: box volume
    density: mass density
    lx,ly,lz: box lengths in x,y,z
    xlo,xhi,ylo,yhi,zlo,zhi: box boundaries
    xy,xz,yz: box tilt for triclinic cells
  </box>

  <lattice docs="lattice">
    xlat,ylat,zlat: lattice spacings in x,y,z
  </lattice>

  <pressure_tensor>
    pxx,pyy,pzz,pxy,pxz,pyz: 6 components of the pressure tensor (pressure units)
  </pressure_tensor>

  <force>
    fmax: max force component on any atom in any dimension
    fnorm: 2-norm (length) of the force vector for all atoms
  </force>

  <crystallographic_cell docs="Howto_triclinic">
    cella,cellb,cellc: periodic cell lattice constants a,b,c
    cellalpha,cellbeta,cellgamma: periodic cell angles alpha,beta,gamma
  </crystallographic_cell>

  <caveats>
  - These keywords are *formulas*, not snapshots. `variable v equal vol` re-evaluates vol each time v is referenced. To capture a value at a specific point, use `variable v0 equal ${v}` (immediate substitution).
  </caveats>

</predefined_variables>
</LAMMPS script parsing rules>

<thermo and barostats>
Thermostat Damping (Tdamp)
For Tdamp, a default value is 100 × timestep. To compute it correctly regardless of the unit system, use LAMMPS variable substitution: Tdamp = $(100.0*dt)
Similarly for Pdamp, a default value is 1000 × timestep.

```lammps.in
fix 1 all nvt temp 298.0 298.0 $(100.0*dt)

# For the barostat,
fix 2 all npt temp 298.0 298.0 $(100.0*dt) iso 1.0 1.0 $(1000.0*dt)
```
</thermo and barostats>

<dump>
Use DCD as the default trajectory format. The only relevant dump_modify option is unwrap (defaults to `no`).
```lammps.in
dump traj all dcd 100 traj.dcd
dump_modify traj unwrap yes
```
</dump>

<restart file>
First, to restart or continue a simulation correctly (for element order and other conditions), you must check both the LAMMPS input file and the lammps-data file of the previous LAMMPS run.
If a subtask is defined as a continuation of a prior LAMMPS run, the input script must begin with the following:

```lammps.in
read_restart {PATH_TO_PREVIOUS_SUBTASK}/{filename}.restart
reset_timestep 0
```
This restores the system from the previous simulation. 
Always put the `reset_timestep 0` command after `read_restart` to reset the current simulation step number to 0, unless there is mention of resetting the current step number. This prevents confusion in later analysis.

Be aware of the following technical requirements:
The list of information included in a restart file, which means these quantities do not need to be re-specified in the input script that reads the restart file, though you can redefine many of these settings:

<included>
units
atom_style and atom_modify settings
timestep size and current step number
simulation box and boundary conditions
atom groups
per-type atom settings (e.g., masses)
per-atom attributes (positions, velocities, bonds, etc.) 
</included>

<not_included>
fix, compute, and variable commands
region definitions
thermo, dump, and restart output settings
pair_style and pair_coeff commands
</not_included>

Lastly, always include the following at the end of the LAMMPS input script, regardless of whether continuation is explicitly mentioned:

```lammps.in
write_restart {filename}.restart
write_data {filename}.lammps-data
```
This command saves the full state of the system in a file named {filename}.restart and atomis structure data, enabling subsequent simulations to continue from where this one ends.
</restart file>
